    Performance Pointers

    Following are some suggestions to get the best performance from Blazix:
    • If using server-based clustering, make sure all cluster servers are operational.  Blazix web-servers will lose performance if they are continually trying to connect to a cluster server.
    • Set-up EJBs to use the IIOP protocol instead of the JRMP default.
    • Set-up EJB data cacheing in the most appropriate manner for your application.  This is a tradeoff between performance and reliability, and many applications may be able to get better performance by changing the default cacheing behavior (default is "on" for loads, "save if changed" for stores.  If cacheing being "on" is acceptable for stores, this may have significant performance improvement.)
    • Minimize Java synchronizations.  If there are multiple synchronizations in a class, use separate lock objects rather than "this" if the synchronizations are unrelated.
    • When the web-site files are no longer changing, pack them in a WAR file using the Archive Packer tool, instead of keeping them as separate files.  Note that the "default" application can also be set-up to run from a WAR file.
    • Make sure entity EJBs are backed by a fast database.  Database performance is a limiting factor in Entity Bean performance.
    • Do not unnecessarily flush or close "out" in JSP files.  If output has not been flushed and the total data length is available at connection commit time, Blazix will be able to honor the Netscape browsers' "Connection: keep-alive" request, thereby improving performance.
